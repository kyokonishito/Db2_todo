## 発生した問題の概要

### 問題1: `pool.init()` のコールバック問題
- **現象**: 30秒タイムアウトが発生
- **原因**: `pool.init()` にコールバックを渡したが、呼ばれなかった
- **影響**: サーバーが起動しない

### 問題2: サーバー再起動ループ
- **現象**: `--watch` モードでサーバーが連続再起動
- **原因**: トップレベルで `await pool.init()` を実行
- **影響**: 開発作業ができない

### 問題3: ESM環境でのJestモック失敗
- **現象**: `jest.mock()` が動作しない
- **原因**: ES Modules環境での不適切なモック方法
- **影響**: テストが実行できない

## 根本原因分析

### 🔍 根本原因1: **ドキュメントへの過信と実装確認の不足**

#### 問題の本質

```
公式ドキュメント記載:
  pool.init(minSize, connStr)

実装した内容:
  pool.init(minSize, connStr, (err) => {
    // このコールバックは呼ばれない
  })

なぜ間違えたか:
  - コールバックパラメータの有無が明記されていなかった
  - 他のNode.jsライブラリの経験から「非同期処理にはコールバックがある」と思い込んだ
  - ドキュメントを読んだだけで、実際の動作確認をしなかった
```

#### 発生メカニズム

```
1. ドキュメントを読む
   ↓
2. 「こうあるべき」という思い込みで実装
   ↓
3. 実際の動作確認をせずに次の機能を実装
   ↓
4. 全機能実装後にテスト
   ↓
5. 問題発生（どこが原因か特定困難）
```

#### 教訓

> 📖 **ドキュメントは参考情報であり、実装の保証ではない**

**なぜこの教訓が重要か**:
- ドキュメントは完璧ではない（記載漏れ、古い情報、曖昧な表現）
- ドキュメントと実装が乖離している場合がある
- 実際の動作が唯一の真実

**具体的な対策**:
1. ドキュメントを読む
2. **小さなテストスクリプトで動作確認**（←これが欠けていた）
3. 動作を理解してから本実装
4. 本実装後も動作確認

---

### 🔍 根本原因2: **他のDBドライバの知識による思い込み**

#### 問題の本質

```
PostgreSQL (pg) での経験:
  // トップレベルで初期化（これが標準パターン）
  export const pool = new Pool({ ... });
  // すぐに使える
  const result = await pool.query('SELECT ...');

Db2 (ibm_db) で同じパターンを適用:
  // トップレベルで初期化
  export const pool = new ibm_db.Pool();
  await pool.init(MIN_CONNECTIONS, connStr);  // ← これが問題

なぜ間違えたか:
  - 「プール初期化はトップレベルで行うもの」という固定観念
  - Node.jsの一般的なパターン（モジュール読み込み時の初期化）を盲目的に適用
  - --watch モードでの動作を考慮しなかった
```

#### 発生メカニズム

```
1. 過去の成功体験（PostgreSQLでうまくいった）
   ↓
2. 「同じパターンが使えるはず」という思い込み
   ↓
3. 新しいライブラリの特性を調査せずに実装
   ↓
4. 開発モード（--watch）で問題発生
   ↓
5. 「なぜ動かないのか」の原因特定に時間がかかる
```

#### 教訓

> 🧠 **過去の経験は参考にするが、新しいライブラリには新しいアプローチが必要**

**なぜこの教訓が重要か**:
- 各ライブラリには独自の設計思想がある
- 「A で動いたから B でも動く」は危険な思い込み
- 環境（開発/本番、watch モード等）によって動作が変わる

**具体的な対策**:
1. 新しいライブラリを使う時は「初心者の心」で臨む
2. 「他のライブラリではこうだった」という思い込みを意識的に捨てる
3. 公式のサンプルコードやデモアプリを確認
4. 開発モード（`--watch`）での動作も必ず確認

---

### 🔍 根本原因3: **テストの不十分さ（モックのみで実DBテストなし）**

#### 問題の本質

```
実施したテスト:
  ✅ モックテスト（14/14 成功）
  ❌ 実DBテスト（未実施）

結果:
  - モックテストは全て成功
  - 実DBに接続すると30秒タイムアウト

なぜ問題が見逃されたか:
  - モックは「期待する動作」をシミュレートするだけ
  - モックは実際のAPIの動作を保証しない
  - pool.init() のコールバック問題はモックでは検出不可能
```

#### モックテストの限界

```javascript
// モックテスト
const executeQuery = jest.fn();
executeQuery.mockResolvedValue([{ ID: 1, TITLE: 'Test' }]);

// このテストは成功するが...
expect(await executeQuery('SELECT ...')).toEqual([{ ID: 1, TITLE: 'Test' }]);

// 実際のibm_dbでは動かない可能性がある
// - pool.init() のコールバック問題
// - conn.close() の呼び忘れ
// - エラーハンドリングの不備
```

#### 教訓

> 🧪 **モックテストと実DBテストは両方必要**

**なぜこの教訓が重要か**:
- モックテスト: ロジックの正しさを検証（単体テスト）
- 実DBテスト: APIの互換性を検証（統合テスト）
- モックが通っても実DBで動かない場合がある

**具体的な対策**:
1. モックテストを実装（ロジック検証）
2. **実DBテストも必ず実装**（API検証）
3. 実装完了の定義に「実DBテスト成功」を含める
4. CI/CDパイプラインに実DBテストを組み込む

---

### 🔍 根本原因4: **段階的な実装・検証の欠如**

#### 問題の本質

```
実際の実装順序:
  1. プール初期化実装
  2. 全CRUD API実装
  3. テストコード実装
  4. 全てまとめてテスト ← ここで問題発覚

問題点:
  - 複数の機能を一度に実装
  - 問題が発生した時、どこが原因か特定困難
  - デバッグに時間がかかる
```

#### 理想的な実装順序

```
✅ あるべき実装順序:
  1. DB接続テスト
     → test-connection.js で確認
     → ✅ 成功を確認してから次へ
  
  2. プール初期化
     → 初期化ロジックのみ実装
     → サーバー起動確認
     → ✅ 成功を確認してから次へ
  
  3. 1つのCRUD操作（GET）
     → GET /api/todos のみ実装
     → curl でテスト
     → ✅ 成功を確認してから次へ
  
  4. 残りのCRUD操作
     → POST, PUT, DELETE を順次実装
     → 各操作ごとにテスト
     → ✅ 全て成功を確認してから次へ
  
  5. モックテスト
     → テストコード実装
     → npm test で確認
     → ✅ 成功を確認してから次へ
  
  6. 実DBテスト
     → 実際のDBで全操作確認
     → ✅ 成功したら完了
```

#### 教訓

> 🔨 **小さく作って、小さく確認する（インクリメンタル開発）**

**なぜこの教訓が重要か**:
- 問題が発生したら、その場で解決できる
- 原因の特定が容易
- 手戻りが少ない
- 進捗が可視化される

**具体的な対策**:
1. 機能を小さな単位に分割
2. 1つの機能を実装したら、すぐに動作確認
3. 問題が発生したら、その場で解決
4. 次の機能に進む前に、現在の機能が完全に動作することを確認

## 再発防止策

### 🛡️ 防止策1: **実装前の動作確認プロトタイプ作成**

#### 実施内容

本実装の前に、小さなテストスクリプトで動作を確認する。

#### 具体例

```javascript
// test-pool-init.js
// 目的: pool.init() の動作を確認

import ibm_db from 'ibm_db';

const connStr = 'DATABASE=TESTDB;HOSTNAME=localhost;...';
const pool = new ibm_db.Pool();
pool.setMaxPoolSize(5);

console.log('=== Test 1: pool.init with callback ===');
pool.init(1, connStr, (err) => {
  console.log('Callback called?', err ? `Error: ${err}` : 'Success');
});

console.log('=== Test 2: pool.open ===');
pool.open(connStr, (err, conn) => {
  console.log('pool.open callback called?', err ? `Error: ${err}` : 'Success');
  if (conn) {
    console.log('Connection obtained, closing...');
    conn.close(() => {
      console.log('Connection closed');
    });
  }
});

// 30秒待って結果を確認
setTimeout(() => {
  console.log('=== Test completed ===');
  console.log('Result: Test 1 callback was', 
    'called' /* or 'NOT called' based on observation */);
  process.exit(0);
}, 30000);
```

#### 実施タイミング

- 新しいライブラリを使う時
- ドキュメントが曖昧な時
- 複雑な機能を実装する前

#### 期待される効果

- 実際の動作を理解してから実装できる
- ドキュメントの記載と実装の乖離を早期発見
- 実装時間の短縮（試行錯誤が減る）

## 実装の鉄則

### 🎯 鉄則1: ドキュメントを信じすぎない

```
❌ 悪い例:
  ドキュメントに書いてある
  → そのまま実装
  → 動かない
  → デバッグに時間がかかる

✅ 良い例:
  ドキュメントを読む
  → 小さなテストで動作確認
  → 動作を理解
  → 本実装
  → すぐに動く
```

**理由**: ドキュメントは完璧ではない。実際の動作が唯一の真実。

---

### 🎯 鉄則2: 思い込みを捨てる

```
❌ 悪い例:
  「PostgreSQLではこうだった」
  → 同じパターンを適用
  → Db2では動かない
  → 「なぜ？」と悩む

✅ 良い例:
  「PostgreSQLではこうだった」
  → 「Db2では違うかもしれない」
  → 公式サンプルを確認
  → Db2の方法で実装
  → すぐに動く
```

**理由**: 各ライブラリには独自の設計思想がある。過去の経験は参考程度に。

---

### 🎯 鉄則3: モックだけで満足しない

```
❌ 悪い例:
  モックテスト実装
  → 全テスト成功
  → 「完成！」
  → 実DBでは動かない

✅ 良い例:
  モックテスト実装
  → 全テスト成功
  → 実DBテストも実施
  → 実DBでも成功
  → 「完成！」
```

**理由**: モックは期待する動作をシミュレートするだけ。実DBテストも必須。

---

### 🎯 鉄則4: 小さく作って小さく確認

```
❌ 悪い例:
  全機能を実装
  → まとめてテスト
  → 複数の問題が発生
  → 原因特定に時間がかかる

✅ 良い例:
  1機能を実装
  → すぐにテスト
  → 成功を確認
  → 次の機能へ
  → 問題が発生してもすぐに解決
```

**理由**: 問題が発生したら、その場で解決できる。手戻りが少ない。
